# 场景B产品化补充功能方案（非商业化）

## 1. 目标与边界

### 1.1 目标
- 将当前场景B从“研究脚本集合”升级为“可持续运行的内部评测系统”。
- 保留现有求解器、评估器、提示词实验、敏感度分析能力，不重写核心算法。
- 支持团队成员稳定复现实验、追踪版本、对比结果、排查异常。
- 产品目标明确对齐论文研究目的：**测试LLM对隐私外部性机制的理解能力**，而不是只做终局结果打分。
- 评估框架采用双轨：`Outcome Correctness` 与 `Mechanism Correctness` 分离。

### 1.2 暂不纳入范围
- 付费、计费、套餐、商业SLA。
- 对外多租户交付能力。
- 场景A/C的统一管理（当前只保留场景B）。

---

## 2. 当前基础能力（可直接复用）

- 理论求解器与Ground Truth：已可输出可验证基准。
- 评估模式：静态博弈 + Fictitious Play。
- Prompt Ladder：v0-v5批量实验流程。
- 敏感度分析：3x3参数网格与跨模型对比。
- 结果产物：JSON/CSV/图表较完整。
- 日志能力：LLM调用日志、失败记录、基础修复逻辑。

这些能力已经是“评测内核”，产品化重点应放在“工程包装层”。

---

## 3. 需要补充的功能（按优先级）

## P0：先把系统跑稳、可追踪

### 3.0 机制诊断导向的数据模型（新增最高优先级）
**现状问题**
- 目前系统虽有多种实验，但产品化语义仍偏“任务执行”，机制诊断信号未被统一建模。

**补充功能**
- 在数据模型中显式区分并存储两类指标：
  - Outcome层：social welfare deviation、share rate偏差等
  - Mechanism层：Jaccard（结构一致性）、EAS（方向一致性）、FP稳定性（过程一致性）
- 对每次运行固定三类诊断标签：
  - Prompt依赖型正确
  - 机制稳定型正确
  - 机制幻觉风险（高Jaccard但低/负EAS）

**实现思路**
- `metrics` 表新增 `metric_level` 字段（outcome/mechanism/process）。
- 新增 `diagnosis_summary` 表，统一沉淀诊断结论，供前端直接展示而非二次计算。
- Compare接口默认按机制指标排序，而非只按单一终局误差排序。

### 3.1 统一任务入口（API/服务化）
**现状问题**
- 主要靠 `python run_xxx.py` 手工触发，自动化和协作成本高。

**补充功能**
- 提供统一任务接口：
  - `创建任务`：指定模式（static/fp/prompt/sensitivity）、模型、参数。
  - `查询状态`：排队中/运行中/成功/失败。
  - `获取结果`：返回关键指标与产物路径。

**实现思路**
- 新建轻量服务层（建议 FastAPI）。
- 服务层只负责参数校验和任务编排，计算仍调用现有脚本/函数。
- 使用统一 `RunSpec` 结构，避免不同脚本参数风格不一致。

---

### 3.2 异步任务与队列执行
**现状问题**
- 长任务（FP、敏感度）阻塞，失败重跑依赖人工。

**补充功能**
- 支持后台任务队列：
  - 提交后异步执行。
  - 失败自动重试（可配置次数）。
  - 超时中断与手动取消。

**实现思路**
- 先用进程池 + 本地队列实现最小闭环；后续可切换 Celery/RQ。
- 每个任务固定工作目录，避免并发写冲突。
- 任务状态机标准化：`PENDING -> RUNNING -> SUCCEEDED/FAILED/CANCELED`。

---

### 3.3 结果元数据落库（替代“只看文件夹”）
**现状问题**
- 结果分散在目录，跨批次检索和统计困难。

**补充功能**
- 新增结果索引数据库（SQLite/PostgreSQL均可）：
  - `runs`：任务级信息（参数、版本、状态、耗时）。
  - `metrics`：核心指标（Jaccard、EAS、MAE、收敛轮数等）。
  - `artifacts`：JSON/CSV/图表路径。
  - `diagnosis_summary`：机制诊断摘要（是否出现机制幻觉、是否需FP干预）。

**实现思路**
- 不迁移旧结果，先对新任务做入库。
- 入库逻辑在任务完成后统一抽取，避免分散在各脚本内。
- 以 `run_id` 作为唯一追踪主键，贯穿日志与产物。

---

### 3.4 版本冻结与可复现追踪
**现状问题**
- prompt、solver参数、代码版本可能变化，历史结果可比性不足。

**补充功能**
- 每次任务自动记录：
  - git commit hash
  - prompt版本（v0-v5或自定义模板哈希）
  - GT版本与参数快照
  - evaluator版本号
  - 机制实验类型（prompt ladder / sensitivity / fictitious play）

**实现思路**
- 定义 `ExperimentFingerprint`，写入 `runs` 表与结果JSON头部。
- 报告对比时默认只比较同一 fingerprint 或明确标记“跨版本对比”。

---

## P1：提升研发效率与调试效率

### 3.5 统一配置中心
**现状问题**
- 参数散落在命令行与脚本内，批量实验易出错。

**补充功能**
- 引入统一配置文件（YAML）：
  - 模型清单
  - 任务模板（static/fp/prompt/sensitivity）
  - 默认重试、超时、并发度

**实现思路**
- CLI/API都接受 `config + override` 模式。
- 输出时回写“实际生效配置”，保证可追溯。

---

### 3.6 结果对比视图（最小前端/报告层）
**现状问题**
- 目前更多是文件级查看，不利于快速判断模型优劣和异常点。

**补充功能**
- 提供最小可视化页面或统一HTML报告：
  - 任务列表与状态
  - 单次运行核心指标卡片
  - 跨模型/跨prompt版本对比图
  - 失败任务快速定位入口（直接跳日志）
  - 机制诊断卡片（Outcome好但Mechanism差的告警）

**实现思路**
- 先做静态报告生成（Jinja2 + Plotly），后续再升级为Web控制台。
- 保持“从数据库拉结果”，不直接扫文件夹。

---

### 3.7 标准化错误分类与诊断
**现状问题**
- 失败信息主要在日志文本里，难做系统性排障。

**补充功能**
- 错误码体系：
  - `E_API_TIMEOUT`
  - `E_JSON_PARSE`
  - `E_RATE_LIMIT`
  - `E_CONFIG_INVALID`
  - `E_RUNTIME_EXCEPTION`

**实现思路**
- 任务执行器统一捕获异常并映射错误码。
- 失败任务输出结构化诊断摘要（错误码、失败步骤、建议动作）。

---

## P2：质量保障与长期维护

### 3.8 回归测试基线
**现状问题**
- 功能迭代后容易影响指标一致性，缺自动预警。

**补充功能**
- 建立最小回归集：
  - 固定随机种子
  - 固定1-2个模型（或mock）
  - 固定prompt版本
- 变更后自动比对关键指标阈值。

**实现思路**
- 在 CI 中跑“轻量冒烟 + 指标阈值检查”。
- 重点监控：Jaccard、share_rate、profit、收敛轮数。

---

### 3.9 数据与产物生命周期管理
**现状问题**
- 结果和日志不断累积，后续会影响检索和存储。

**补充功能**
- 产物分层保存策略：
  - 热数据（最近30天）保留完整日志。
  - 温数据（30-180天）保留摘要+关键产物。
  - 冷数据只保留摘要和可复现实验指纹。

**实现思路**
- 定期清理任务（cron）+ 保留白名单run_id。
- 删除前先做摘要固化（防止信息丢失）。

---

## 4. 前端设计补充方案（场景B）

### 4.1 前端目标
- 让研究同学不需要进服务器目录，也能完成“发起评测、查看结果、对比模型、定位失败”的闭环。
- 界面只服务场景B，避免过早做通用平台复杂度。
- 与后端的任务/结果模型保持一一对应，减少前后端语义偏差。
- 前端叙事以“机制理解诊断”为中心：先回答模型是否理解机制，再展示它是否做对结果。

### 4.2 信息架构（IA）
- 一级导航建议固定为4页：
  - `Runs`：任务列表与状态管理。
  - `Run Detail`：单次任务完整详情（参数、指标、图表、日志入口）。
  - `Compare`：跨模型/跨prompt版本/跨参数区间对比（内置机制幻觉象限）。
  - `Configs`：模型配置、任务模板、默认参数。

- 全局检索与过滤维度：
  - 时间区间
  - 模型名
  - 运行模式（static/fp/prompt/sensitivity）
  - 状态（成功/失败/运行中）
  - prompt版本（v0-v5）

### 4.3 页面结构与关键交互

### 4.3.1 Runs（列表页）
- 表格字段建议：
  - run_id
  - mode
  - model
  - prompt_version
  - status
  - duration
  - created_at
- 关键交互：
  - 新建任务（弹窗）：填写RunSpec并提交。
  - 批量重跑失败任务。
  - 一键跳转到失败日志。

### 4.3.2 Run Detail（详情页）
- 分区建议：
  - `概览卡片`：Jaccard、EAS、share_rate误差、profit误差、是否收敛。
  - `机制判定`：是否出现“高Jaccard但低EAS”的机制幻觉。
  - `参数快照`：模型配置、prompt版本、GT版本、commit hash。
  - `过程可视化`：FP轮次变化（分享率、策略稳定性）。
  - `产物区`：JSON/CSV/图表下载。
  - `诊断区`：错误码、失败步骤、重试记录。
- 关键交互：
  - 从结果回放到调用日志（按call_id定位）。
  - 对本次run发起“复制参数并新建任务”。

### 4.3.3 Compare（对比页）
- 对比模式：
  - 同一prompt版本下多模型对比。
  - 同一模型下prompt v0-v5对比。
  - 敏感度3x3网格热力图对比。
- 图表建议：
  - 模型排名条形图
  - 参数网格热力图
  - FP收敛曲线多线图
  - 误差分布箱线图
  - Jaccard-EAS 二维象限图（Strong / Surface Fitting / Potential / Weak）
- 关键交互：
  - 选择“统一fingerprint过滤”以保证可比性。
  - 导出对比报告（HTML/PDF）。

### 4.3.4 Configs（配置页）
- 展示内容：
  - 模型清单（只读展示关键字段）
  - 任务模板（static/fp/prompt/sensitivity）
  - 全局默认值（重试、超时、并发）
- 关键交互：
  - 保存配置新版本并记录变更说明。
  - 对配置版本进行回滚（仅回滚配置，不回滚结果）。

### 4.4 前端状态与异常设计
- 状态标签统一：`PENDING / RUNNING / SUCCEEDED / FAILED / CANCELED`。
- 失败展示最小集合：
  - 错误码
  - 错误摘要
  - 失败时间
  - 重试次数
  - 快速重跑按钮
- 长任务体验：
  - 显示心跳时间和最近阶段（如“正在跑round 17/50”）。
  - 支持用户手动取消。

### 4.5 前后端接口契约（前端视角）
- 最小接口集合：
  - `POST /runs`：创建任务
  - `GET /runs`：分页查询任务
  - `GET /runs/{run_id}`：任务详情
  - `POST /runs/{run_id}/cancel`：取消任务
  - `POST /runs/{run_id}/retry`：重试任务
  - `GET /compare`：对比查询
  - `GET /diagnosis/{run_id}`：机制诊断摘要
  - `GET /artifacts/{artifact_id}`：下载产物
- 契约要求：
  - 所有数值指标提供单位或定义说明（避免前端误解）。
  - 时间统一ISO8601。
  - 错误码与错误消息分字段返回。

### 4.6 视觉与组件规范（简化版）
- 组件优先级：
  - 表格（任务）
  - 指标卡（核心KPI）
  - 图表组件（折线/热力/箱线）
  - 日志查看抽屉
- 视觉原则：
  - 颜色映射固定（成功=绿，失败=红，运行中=蓝，排队=灰）。
  - 图表配色在所有页面保持一致，避免同指标跨页颜色变更。
  - 重要指标支持阈值高亮（如Jaccard低于阈值标黄/红）。

### 4.7 前端实施建议（2阶段）
- 阶段1（快速可用）：
  - 技术栈：React + Ant Design + ECharts（或Plotly）。
  - 完成 Runs + Run Detail 两页，先解决“可管理、可追踪”。
- 阶段2（分析增强）：
  - 增加 Compare + Configs。
  - 增加报告导出与高级过滤。

## 5. 推荐落地顺序（6周）

### 第1-2周（P0核心）
- 完成统一任务API（创建/查询/结果）。
- 建立异步执行器与状态机。
- 跑通 static + fp 两种模式的端到端任务流。
- 建立双轨指标入库（Outcome/Mechanism）。

### 第3-4周（P0补齐）
- 接入结果元数据入库。
- 加入版本指纹（commit/prompt/GT/配置）。
- 实现基础失败重试、取消、超时。
- 实现机制诊断摘要生成（含机制幻觉判定规则）。

### 第5周（P1）
- 完成统一配置中心。
- 产出统一HTML报告（单次 + 对比 + 机制象限）。

### 第6周（P1/P2）
- 建立错误码体系。
- 上线最小回归测试流水线。

---

## 6. 完成标志（DoD）

- 任意场景B任务都可通过API异步提交并追踪状态。
- 任务完成后，核心指标可通过数据库查询，不依赖手工找文件。
- 任意结果可追溯到明确版本指纹（代码/提示词/参数）。
- 失败任务有结构化错误摘要，重试策略可配置。
- 至少有一套自动回归流程，能在代码变更后给出指标偏移预警。
- 前端可完成完整闭环：创建任务 -> 查看过程 -> 对比结果 -> 定位失败。
- 系统可稳定输出机制诊断结论：区分“结果正确”与“机制正确”。
- 对每个模型至少可给出一次可解释的机制幻觉判定（若存在）。

---

## 7. 一句话总结

场景B当前“算法与实验能力”已经够强，产品化重点不是再做新指标，而是补齐**服务层、任务层、数据层、追踪层**，把一次性实验流程变成可持续运行的工程系统。
